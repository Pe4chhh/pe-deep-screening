分析逻辑

本页介绍 PE Deep Screening 从信息检索到最终评分的完整分析链路与决策框架。所有环节均已工程化落地，非概念设计。
一、信息检索管线 — 多源分层 · 结构化去重 · 质量过滤 ▼
层级来源说明
L1 多源搜索Exa / Tavily / 百度按尽调主题（经营、规模、风险、市场等）构造查询词，Exa 优先 → Tavily 兜底 → 百度补充中文源。搜索结果经公司名匹配、垃圾域名过滤、URL 去重
L2 微信补充微信公众号文章CimiData API 搜索微信公众号经营事实（产品、客户、产能），用于交叉验证和补充线索，不作为独立证据来源
L3 工商核验企查查 MCP自动查询工商登记、股东、实控人、经营状态、参保人数等注册信息，作为身份核验和预筛选数据基础
L4 深度抓取Playwright 浏览器对搜索结果中的关键页面（官网、新闻、行业数据库）进行全文抓取，提取结构化经营数据
L5 AI 自主补搜LLM Tool CallingAI 在分析过程中发现信息缺口时，可自主调用 search_web 工具补充搜索（最多 3 轮），搜索结果自动纳入统一编号引用体系
二、评分框架 — Compounder 透镜 · 5 维加权 · 程序复算 ▼

AI 对每个维度输出 1-5 分 + 分析理由 + 证据等级。程序用固定权重复算企业质量分与 ABCD 评级，不相信 AI 的算术。

定性输出：角色定位（Platform / Add-on / Both / Neither）+ 交易可行性（evidence_yes / inferred_yes / inferred_no / evidence_no）
三、决策规则 — ABCD 评级 → 推进建议 · 红旗联动 · 自我审查 ▼
条件企业质量分推进建议
质量高 + 规模适配 + 无障碍≥ 3.8 (A)推进
质量高 + 规模可能不适配≥ 3.0 (A/B)观察
质量一般 / 规模明确不适配2.0–3.7 (B/C)观察
质量差 / 严重红旗 / 交易不可行< 2.0 (C/D)不建议推进

红旗约束：14 项 PE 红旗（一次性收入、大客户依赖、老板关系、技术替代等），严重红旗或 ≥3 项红色 → 强制不建议推进。
交易可行性否决：evidence_no → 强制不建议推进。
自我审查（Reflection）：评分完成后程序自动检查内部矛盾（如 A 级 + 规模明确过小 → 推进），发现矛盾自动修正。

四、批量预筛选 — 8 条表格规则 · 短路执行 · 零 API 消耗 ▼

上传企查查 Excel 后，程序先用表格中的工商数据做硬性排除。被排除的公司不消耗 API 额度，直接跳过完整 AI 深筛。预期筛掉 50-70%。

R5企业类型国有独资/集体所有制/全民所有制
R3企业类型国有/集体/集团/央企体系
五、技术架构 — Agentic RAG · Multi-Model · Guardrails ▼
Agentic RAG：AI 不是被动分析给定文本，而是持有搜索工具，自主决定搜什么、搜几轮、搜完怎么用。程序预搜索提供信息起点，AI 在分析中动态补搜。
Multi-Model Router：支持 DeepSeek V4 Pro（默认，工具调用能力强）、Kimi K2.6（256K 上下文免工具调用）、Moonshot V1 128K（备选）。
LLM-as-Judge + Guardrails：AI 负责观察和推理，程序负责裁判——固定权重重算总分、红旗规则强制封顶维度分、交易可行性硬否决。
Streaming Response：分析阶段实时流式输出文本到前端，用户可知 AI 进度。
Structured Output：10 种容错策略的 JSON 解析器 + 终极正则兜底，确保非结构化 LLM 输出可被程序化处理。
Self-Critique：评分完成后自动检查 A级+规模过小仍推"推进"等内部矛盾，发现即修正。
