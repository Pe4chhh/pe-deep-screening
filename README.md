# PE Deep Screening

AI 驱动的私募股权（PE）并购标的筛选 Agent 系统。给 LLM 装上搜索引擎，让它自主完成从信息搜集到多维评分的完整尽调分析。

**核心洞察**：传统"程序搜索 → 截断 → AI 被动分析"模式，AI 只能分析给定的静态文本。本系统改为 **LLM 持有搜索工具，自主决定搜什么、搜几轮**，程序负责裁判和纠错。

---

## 功能概览

### 模式一：单个深筛

输入一家公司名称 → 五层搜索管线 + AI 工具调用自主补搜 → 5 维评分 + 14 项红旗检查 + 角色定位 + 交易可行性 → 输出 HTML 结构化报告。

实时流式展示 AI 分析文本，网页进度条显示 4 阶段（搜索 → 抓取 → 分析 → 生成）。报告包含雷达图、评分卡片、红旗清单、参考来源链接，可直接分享 URL。

### 模式二：批量预筛选 + 深筛（两步式）

1. **预筛选**：上传企查查 Excel → 8 条硬性规则秒级执行（经营状态、地域、成立年限、参保、注册资本、企业类型）→ 纯 Python 判断，零 API 消耗。预计筛掉 50-70%。
2. **深筛**：用户查看排除结果后，手动启动 → 仅对通过公司执行完整 AI 分析。每步均可单独下载 Excel。

### 模式三：分析逻辑说明页

内置"📖 分析逻辑"标签页，用可折叠面板展示完整的检索管线、评分框架、决策规则、预筛选规则和技术架构，便于外部 reviewer 理解系统。

### 历史报告管理

分页浏览所有历史深筛报告 + 批量任务记录 + Tavily API 额度监控。支持删除、翻页、按评级/推进建议筛选统计。

## 技术架构

| 层级 | 技术选型 |
|------|---------|
| **Agent 引擎** | LLM Function Calling（search_web / fetch_page），最多 3 轮自主补搜 |
| **搜索管线** | Exa → Tavily → 百度（中文源）→ 微信公众号（CimiData API）→ Playwright 浏览器全文抓取 |
| **工商数据** | 企查查 MCP 协议集成，6 个模块（公司 / 风险 / 知识产权 / 经营 / 高管 / 历史）|
| **多模型路由** | DeepSeek V4 Pro（默认，工具调用）/ DeepSeek V4 Flash / Kimi K2.6（256K 上下文）/ Moonshot V1 128K |
| **评分引擎** | Compounder 透镜 5 维加权：RR 30% + 市场地位 25% + 商业模式 20% + 客户粘性 15% + 行业匹配度 10% |
| **Guardrails** | AI 打分 → 程序固定公式重算 → 14 项 PE 红旗规则强制封顶 → 交易可行性硬否决 → 自我审查（Reflection）|
| **Structured Output** | `---JSON---` 模板 + 10 种容错解析策略 + 终极 regex 字段提取兜底 |
| **流式响应** | SSE streaming，分析阶段实时推送 AI 文本到前端滚动展示 |
| **前端** | 原生 HTML/CSS/JS，FastAPI (Python) 后端，Nginx 反向代理，systemd 管理 |

## 工程亮点

### Agentic RAG

不是传统的"搜一次 → 拼进 prompt → 输出"。LLM 在分析过程中发现信息缺口时**自主调用搜索工具**，搜索结果自动纳入统一编号引用体系。程序预搜索提供信息起点（5 个固定尽调主题），AI 在此基础上动态补搜。搜索结果经垃圾域名过滤、URL 去重、公司名匹配后才喂给 AI。

### LLM-as-Judge + Guardrails

核心设计原则：**AI 负责观察和推理，程序负责裁判**。

- 每个维度 AI 输出 1-5 分 + 分析理由 + 证据等级（4=直接证据 / 3=较强公开证据 / 2=合理推断 / 1=未见公开依据）
- 程序用固定权重（不是 AI 声称的权重）重算企业质量分和 ABCD 评级
- 14 项 PE 红旗触发时，强制对关联维度分封顶（如"依赖单一大客户 🔴"→ 客户粘性 ≤ 2 分）
- 交易可行性 `evidence_no` → 强制不建议推进（硬否决）
- 评分完成后自动运行自我审查：A 级 + 规模明确过小仍推"推进"？自动修正为"观察"

### 10 层 JSON 解析容错

LLM 输出结构化 JSON 是 Agent 开发的经典工程难题。本系统实现：

1. `json` 代码块提取
2. 直接解析
3. 首尾大括号截取
4. JS 注释 / 尾部逗号 / 单引号修复
5. 控制字符转义
6. 嵌入双引号智能替换为中文引号
7. json5 宽松解析
8. 正则逐字段提取（终极兜底，提取 30+ 字段并补齐默认值）

所有策略失败时，正则提取核心字段补齐默认值，**绝不因 JSON 损坏而丢弃整条分析**。

## Prompt 迭代 & 调试历程

这是本项目最耗费精力的部分。以下记录了关键问题及解决：

| 问题 | 现象 | 根因 | 解决方案 |
|------|------|------|---------|
| 评分不一致 | 同一公司多次分析，评级在 A/B/C 间跳变，推进建议三种结果都出现过 | LLM 随机性 + 搜索 API 每次返回不同结果 + prompt 版本迭代 | `temperature=0` 消除 LLM 随机性；统一评分标准措辞 |
| 章节内容串位 | sec2 内容出现在 sec3，报告卡片错乱 | 6000 字超长 prompt 中维度顺序与 JSON 字段编号不一致 | 在 JSON 模板中显式标注 `sec2 = 商业模式（非 RR）` 映射关系 |
| 假阳性交易障碍 | "经核实未见交易障碍"被判定为"存在交易障碍" | 自由文本中误匹配了"交易障碍"关键词 | 改为 `has_transaction_obstacle` 布尔字段，程序判断不依赖文本匹配 |
| AI 给出乐观误判 | 非 PE 标的（鲜切水果配送）被评 A 级推"推进" | LLM 受正向关键词影响，缺乏领域边界判断 | 增加行业匹配度维度（10%权重）+ 硬编码行业列表 |
| JSON 解析失败 | 完整分析报告卡片全部为空 | LLM 在 JSON 字符串值内嵌入未转义的双引号 | 加"嵌入引号修复"策略 + 终极 regex 字段提取兜底 |
| 批量分析假死 | 某公司卡住，整个 batch 永远停在"预筛选中" | asyncio.create_task 静默吞异常，timezone 未导入导致 NameError | 添加 per-company 10 分钟超时 + 顶层 try-except + 导入修复 |
| 规模推断矛盾 | 同一公司营收推断区间 500 万 → 1 亿，波动 20 倍 | 搜索引擎每次返回不同文章（提到营收 vs 没提到） | 增加 `revenue_evidence` 等级标注，低证据不据此硬排 |
| 批量评分与单个不一致 | 批量用简版 prompt（5 维），单个用完整版（5 维 + 定性），导致评分标准不同 | 两套 prompt 字段规格不一致 | 统一为同一套 JSON 模板 |
| 企查查重复调用 | 每次分析调用 6 次企查查 MCP × 2（调用两次） | 代码中 `collect_qcc_evidence()` 被调了两处 | 删除冗余调用，合并为一次 |

## 批量预筛选规则（8 条）

| 规则 | 检查项 | 判定 | 数据来源 |
|------|--------|------|---------|
| R8 | 经营状态 | 含注销 / 吊销 / 清算 / 停业 / 异常 / 违法 | 企查查工商列 |
| R1 | 省份 | 不在目标地域（沪苏浙皖闽湘鄂粤川渝）| 企查查工商列 |
| R7 | 成立日期 | 距今 < 3 年 | 企查查工商列 |
| R9 | 参保人数 | ≤ 5 人 | 企查查工商列 |
| R11 | 注册资本 | < 50 万且实缴为 0 | 企查查工商列 |
| R12* | 外资控股 | 企业类型含外商 / 港澳台 / 中外合资（可选，默认不启用）| 企查查工商列 |
| R5 | 国资控股 | 国有独资 / 集体所有制 / 全民所有制 | 企查查工商列 |
| R3 | 集团体系 | 含国有 / 集体 / 集团 / 央企 | 企查查工商列 |

搜索型规则 R2（上市公司子公司）/ R4（已上市）/ R6（近期 VC/PE 融资）保留到 AI 分析阶段判断。

## 运行

```bash
# 环境变量
export DEEPSEEK_API_KEY=sk-xxx
export DEEPSEEK_BASE_URL=https://api.deepseek.com
export DEEPSEEK_MODEL=deepseek-v4-pro
export KIMI_API_KEY=sk-xxx       # 可选
export TAVILY_API_KEY=tvly-xxx   # 可选
export QCC_MCP_KEY=xxx           # 可选

# 启动
cd pe-screening-web
pip install -r requirements.txt
python server.py
# → http://localhost:8000
```

生产部署：systemd 托管 uvicorn，Nginx 反向代理 + SSL，Cloudflare DNS。

## 已知局限 & 下一步

- **搜索不确定性**：搜索 API 每次返回不同结果，是系统性风险。已通过 `temperature=0` 消除 LLM 侧变量。
- **缺少自动化评测集**：改 prompt 后依靠人工跑案例验证。计划建立 30 家"黄金标准"公司评测集。
- **单文件架构**：`server.py` 承载过多职责，后续拆分为 `engine/llm.py` / `engine/search.py` / `engine/scoring.py`。
- **Agent trace 缺失**：工具调用仅通过 print 输出，缺少结构化决策日志用于调试。
